# Design Tradeoffs & Decisions

Technical rationale for non-obvious choices made throughout the pipeline.
Intended audience: a code reviewer or interviewer asking "why did you do it this way?"

---

## 1. Random Forest over XGBoost as the "best" model

**Decision:** Named RF (PR-AUC 0.820, F1 0.857) the best model, not XGBoost (PR-AUC 0.794, F1 0.809).

**Rationale:** On this dataset, RF outperforms XGBoost on every evaluation metric. XGBoost's main advantage — faster training — matters more in production than in a batch portfolio pipeline. If I were deploying this for real-time scoring (sub-100ms latency requirement), XGBoost would be the production choice: it trains in 2.8s vs. RF's 26s, and can be serialized to a smaller file. The portfolio deliberately surfaces this tradeoff rather than silently picking one.

**What I'd change in production:** XGBoost with further hyperparameter tuning (learning rate decay, max_depth search, early stopping on a validation set) would likely close the gap. RF's ensemble of 200 trees is also slower at inference time than a tuned XGBoost model.

---

## 2. Class weighting over SMOTE as the primary imbalance strategy

**Decision:** The best model (RF) uses `class_weight='balanced'` rather than SMOTE oversampling.

**Rationale:** For the LR experiments, SMOTE produced the lowest cost ($7,825 vs. $8,805 for class weighting) but at the cost of 1 minute of resampling on 227k rows. For tree-based models, class weighting is built into the splitting criterion and is effectively free. SMOTE's risk on this dataset is that synthetic fraud samples are generated by interpolating between real fraud cases — but with only 417 training fraud cases, the interpolation space is small and synthetic samples can be implausible.

**When SMOTE is the right call:** When the model is a neural network or SVM that doesn't support class weighting natively, or when the class imbalance is so extreme (>1000:1) that weighting alone is insufficient.

## 3. PR-AUC over ROC-AUC as the primary metric

**Decision:** Used Precision-Recall AUC throughout, not ROC-AUC.

**Rationale:** At a 0.173% fraud rate, ROC-AUC is deeply misleading. A model that assigns a random score between 0 and 1 to every transaction achieves ~0.5 ROC-AUC (correctly useless). A model that predicts "legit" for every transaction except the most obvious fraud cases can still achieve 0.95+ ROC-AUC because the true-negative rate (specificity) is nearly perfect by default — there are 578 legitimate transactions for every fraudulent one. PR-AUC doesn't get this "free ride" from the majority class because it never considers true negatives.

**Why not plain F1:** F1 at a single threshold (e.g., 0.5) is threshold-sensitive and doesn't reflect model calibration. PR-AUC summarizes performance across all thresholds, which is the right summary for a model whose operating threshold will be tuned by a business team.

## 4. Cost-weighted metric instead of F-beta

**Decision:** Used `total_cost = FN × $500 + FP × $5` rather than F-beta with a fixed beta.

**Rationale:** F-beta requires choosing beta (how many times worse is a false negative than a false positive?). The answer is domain-specific and changes as cost assumptions change. A dollar-denominated cost function is more interpretable to a business stakeholder ("the model costs $9,005 per 10-hour period") and makes it easy to re-run the analysis when cost assumptions change. The dashboard lets analysts adjust FN and FP costs interactively to see how the optimal threshold shifts.

**Assumptions documented:**
- FN cost: $500 — conservative fraud loss estimate (average fraud in this dataset is ~$122, but bank chargebacks, ops overhead, and reputational cost push this higher)
- FP cost: $5 — analyst review time (~15 min at $20/hr) + marginal customer friction

## 5. Time-based train/test split (not random)

**Decision:** Split at the 80th percentile of `time_seconds` (first ~38 hours train, last ~10 hours test) rather than a random shuffle.

**Rationale:** Credit card transactions are a time series. Randomly shuffling and splitting would allow the model to train on transactions that occurred *after* transactions it is then evaluated on — a form of data leakage. A time-based split simulates the real deployment scenario: the model is trained on historical data and evaluated on future transactions it has never seen.

**Practical consequence:** The test set has a slightly lower fraud rate (0.132%) than the training set (0.183%), which is realistic — fraud patterns shift over time and a model trained on older data degrades on newer data.

## 6. Velocity features in pandas rather than SQL

**Decision:** Computed rolling velocity features (`txn_count_last_1hr`, etc.) in pandas rather than using `RANGE BETWEEN N PRECEDING AND CURRENT ROW` SQL window functions.

**Rationale:** Multiple `RANGE BETWEEN` window functions on a single 284k-row table, each requiring Postgres to sort the full dataset and scan backwards, exceeded 1 hour of runtime before being killed. Pandas `rolling()` on a sorted numeric index computes the same semantics in under 1 second by maintaining a sliding pointer rather than re-scanning.

**The SQL is still documented:** `sql/feature_engineering.sql` shows the equivalent window function definitions for reference. In a production data warehouse with proper partitioning by `card_id` and columnar storage (Redshift, BigQuery, Snowflake), the SQL approach would be preferred since it operates close to the data without a Python round-trip.

**Root limitation:** Without a `card_id` column (V1–V28 are PCA-anonymized), velocity features can only be computed over the global transaction stream — they don't represent per-card velocity, which is what a real fraud team would use. The SQL pattern is production-correct; the missing partition key is a data limitation, not an engineering one.

---

## 7. No hyperparameter tuning (deliberate)

**Decision:** Used reasonable default hyperparameters for RF and XGBoost rather than running grid search or Bayesian optimization.

**Rationale:** Hyperparameter tuning on an imbalanced dataset requires careful stratified cross-validation and a PR-AUC scoring function — adding several hours of compute for marginal gains on a portfolio project. The goal was to demonstrate the *pipeline* (data → features → model → evaluation → dashboard) rather than squeeze the last 0.5% of PR-AUC. A production system would warrant tuning, and the `train.py` structure makes it straightforward to add `GridSearchCV` or `optuna` to any experiment block.

---

## 8. Streamlit over Tableau

**Decision:** Built the dashboard in Streamlit rather than Tableau Public.

**Rationale:** Streamlit allows the threshold slider to recompute all metrics live without a data export/refresh cycle. Tableau's parameter actions can approximate this but require pre-aggregated data at every threshold value. For a live precision/recall/cost curve tied to a draggable slider — the centerpiece of this dashboard — Streamlit is unambiguously the right tool. Tableau would be preferable for a fixed executive-facing report with richer formatting.
